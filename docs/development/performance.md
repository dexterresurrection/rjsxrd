# Оптимизации и производительность

## Параллелизм

| Операция | Механизм | Потоков |
|----------|----------|---------|
| Загрузка URL | ThreadPoolExecutor | **50** (настраивается) |
| Запись файлов | ThreadPoolExecutor | 8 |
| Фильтрация конфигов | ProcessPoolExecutor | 8* |
| Xray-тестирование (Linux) | Async + curl_cffi | **300** (ранее 150, увеличено после фикса ETA, ~50MB RAM на процесс) |
| Xray-тестирование (Windows) | ThreadPoolExecutor | **50** |
| TCP-пинг (Linux) | asyncio | до 300 |
| TCP-пинг (Windows) | asyncio | до 150 |

\* Количество CPU.

**Платформенная оптимизация:** Windows имеет более высокие накладные расходы на процессы, поэтому конкурентность ниже. Linux/WSL может обрабатывать больше параллельных операций.

### ExecutorCache — кэширование пулов потоков

Все ThreadPoolExecutor'ы проходят через `ExecutorCache` (`utils/executor_cache.py`):

- **lazy-инициализация** — пулы создаются при первом использовании, переиспользуются во всех последующих вызовах
- **WSL-детекция** — автоматическое снижение workers на WSL (до `default/2`) для предотвращения утечки памяти vmmem
- **Адаптация по памяти:**
  - `< 2 GB`: 2 workers
  - `2-4 GB`: 4 workers
  - `> 4 GB`: полное количество
- **atexit-завершение** — все пулы корректно закрываются при выходе

Типы исполнителей:

| Имя | Workers | Назначение |
|-----|---------|------------|
| `file_io` | 8 | Параллельная запись chunk'ов |
| `network_io` | 50 | Загрузка URL |
| `cpu_bound` | CPU count | Фильтрация через ProcessPoolExecutor |
| `regex` | 8 | `filter_secure_configs()`, SNI/CIDR |

**Процессы vs Потоки:**
- `filter_secure_configs()` использует ProcessPoolExecutor (CPU-bound — проверка хэшей, base64, JSON)
- Остальные операции — ThreadPoolExecutor (I/O-bound — сеть, диски, regex под GIL)

Переменные окружения для тюнинга:

```bash
ASYNC_CONCURRENCY_WIN32=100   # Windows (по умолчанию 50)
ASYNC_CONCURRENCY_LINUX=300   # Linux (по умолчанию 300)
VALIDATION_TCP_CONCURRENCY=200
FETCH_TIMEOUT=4               # Таймаут HTTP запроса (по умолчанию 5)
FETCH_MAX_ATTEMPTS=1          # Попыток загрузки URL (по умолчанию 2)
MAX_WORKERS=80                # Параллельных загрузок URL (по умолчанию 50)
```

## Кэширование

- **DNS:** 60-секундный TTL, lock-free, aiodns-резолвер
- **Host/port:** `lru_cache` на `extract_host_port()` — избегает повторного парсинга URL
- **HTTP:** Connection pooling в curl_cffi-сессиях (keep-alive)

## Сетевые оптимизации

- **curl_cffi** вместо `requests` — в 2-3x быстрее, TLS-impersonation (Chrome 124), обход анти-бот систем
- **SOCKS5** формат `socks5://` для curl_cffi
- **Remote DNS** через `socks5h://` — предотвращает DNS-утечки
- **Chrome User-Agent** для совместимости с источниками

## Управление ресурсами

- **Порты:** динамическое выделение, проверка доступности, диапазоны 20000-24999
- **Процессы:** signal handlers + atexit + psutil (гарантированная очистка)
- **Логи:** агрессивная фильтрация спама из логов Xray
- **Временные файлы:** `tempfile.mkstemp()` + `chmod 0600` (только владелец)
- **Таймауты:** `VALIDATION_TCP_TIMEOUT=3s`, `VALIDATION_HTTP_TIMEOUT=5s`, `XRAY_STARTUP_TIMEOUT=3s`
- **Circuit breaker:** 5 ошибок подряд → 60 с восстановления (`CIRCUIT_BREAKER_FAILURE_THRESHOLD=5`, `CIRCUIT_BREAKER_RECOVERY_TIMEOUT=60`)
- **Retry:** 2 попытки на конфиг (Xray), 5 попыток на загрузку в GitHub (экспоненциальная задержка от 0.5 с)

### DNS-кэш

`DNS_CACHE_TTL_SECONDS=60` — lock-free реализация через `aiodns`. Без блокировок, 60-секундный TTL для динамических IP.

### Порты Xray

| Диапазон | Назначение |
|----------|------------|
| 20000-21999 | Batch-тестирование (2000 портов) |
| 22000-23999 | Прокси-цепочки (dialerProxy) |
| 24000-24999 | Персистентные прокси |

`XRAY_PORT_RANGE=5000` — увеличен с 1000 для избежания TIME_WAIT конфликтов. `XRAY_PORT_MAX_ATTEMPTS=10` попыток на свободный порт.

## SmartETA

Умный оценщик времени для batch-тестирования.

**Проблема:** быстрые конфиги завершаются первыми, глобальная средняя скорость завышена, стандартный ETA постоянно показывает меньше реального времени.

**Решение:**
1. **Трёхкомпонентная оценка** — макс из трёх: скользящее окно (быстрая реакция), глобальная скорость (стабильная), EMA длительности конфигов (не зависит от очередности)
2. **EMA длительности** — экспоненциальное скользящее среднее, обновляется на каждом конфиге (не только на batch). Адаптируется к текущему распределению быстрых/медленных
3. **Timeout floor** — физическое ограничение: `ceil(remaining / concurrency) * timeout` (без дисконта)
4. **Динамическое окно** — размер `max(500, min(total // 10, 5000))`, масштабируется под 30k-100k конфигов
