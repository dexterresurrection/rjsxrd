# Система безопасности и фильтрации

Проект реализует многоуровневую систему фильтрации небезопасных конфигов и генерацию наборов для обхода мобильных белых списков.

## Фильтрация insecure-конфигов

Функция `has_insecure_setting()` проверяет каждый конфиг на наличие параметров, снижающих безопасность. Используется для генерации `-secure.txt` файлов.

### VMess

| Проверка | Параметр | Причина |
|----------|----------|---------|
| Отключение TLS | `security=none` | Трафик не шифруется |
| Принудительный insecure | `insecure=true`, `allowInsecure=true` | Сертификат не проверяется |
| Устаревший режим | `alterId`/`aid` > 0 | Уязвимость MD5 (VMess legacy) |

Парсер также валидирует: TLS требует непустой SNI, h2 host должен быть корректным массивом.

### VLESS

| Проверка | Параметр | Причина |
|----------|----------|---------|
| Без шифрования | `encryption=none` (без TLS/Reality) | Трафик открыт |
| Insecure | `allowInsecure=true`, `insecure=true` | TLS-сертификат не проверяется |

Парсер Reality: требует одновременного наличия `sni` и `publicKey`.

### Trojan

| Проверка | Параметр |
|----------|----------|
| Insecure | `allowInsecure=true`, `insecure=true` |

Парсер Reality: те же требования, что у VLESS.

### Shadowsocks

| Проверка | Критерий |
|----------|----------|
| Слабый шифр | Единый список в `security_filter.py` — `rc4`, `des`, `rc4-md5`, все `aes-*-cfb`/`-cfb8`/`-cfb1`/`-ctr`, `bf-cfb`, `camellia-*-cfb`, `cast5-cfb`, `des-cfb`, `idea-cfb`, `rc2-cfb`, `seed-cfb`, `salsa20`, `chacha20`, `xsalsa20`, `xchacha20` и др. |
| Пустой пароль | Длина пароля = 0 |
| Плагин | Любой plugin-параметр (не поддерживаются) |
| **2022 длина ключа** | Base64-ключ для `2022-blake3-aes-128-gcm` должен декодиться в 16 байт, для `2022-blake3-aes-256-gcm` и `2022-blake3-chacha20-poly1305` — в 32 байта. Multi-key формат `key1:key2` (3x-ui) пропускается — это фича Xray-core, не ошибка. |

**Безопасные шифры (AEAD):**
```
aes-128-gcm, aes-256-gcm, chacha20-ietf-poly1305,
chacha20-poly1305, xchacha20-ietf-poly1305,
2022-blake3-aes-128-gcm, 2022-blake3-aes-256-gcm,
2022-blake3-chacha20-poly1305
```

Константы определены в `source/utils/security_filter.py:11-29` — единый источник истины, импортируется в `xray_tester.py` и `protocol_parsers.py`.

### ShadowsocksR

Конвертируется в Shadowsocks (протокол и обфускация SSR теряются). Используется та же валидация шифров и пустых паролей.

### TUIC

Не поддерживается Xray-core — парсер возвращает `None`. Параметр `skip-cert-verify` считается небезопасным.

### Hysteria

- Hysteria v2: валидация TLS SNI
- Hysteria v1: предупреждение при `insecure=1`

### Общие проверки

```
verify=0 | verify=false | insecure=1/true/yes/on
```

> **Важно:** `verify=0` проверяется как отдельный параметр URL (regex `[\?&]verify=0`), а не простой substring. Это предотвращает ложные срабатывания (например, `skip-cert-verify=0` не срабатывает как insecure).

### Проверки без TLS

Для VLESS без TLS/Reality: если конфиг `vless://` не содержит ни `security=tls`, ни `security=reality`, он считается небезопасным (plaintext трафик).

## SNI/CIDR-фильтрация

Предназначена для обхода мобильных белых списков (Россия, Узбекистан и др.), где операторы блокируют VPN по SNI и IP-адресам.

**Принцип:** конфиги, направленные на «белые» домены/IP (Avito, Yandex, Mail.ru и т.д.), не блокируются мобильным оператором, так как выглядят как обычный трафик на разрешённые ресурсы.

**Процесс:**
1. Загрузка доменов из `config/whitelist-all.txt` (сотни доменов)
2. Загрузка CIDR-диапазонов из `config/cidrwhitelist.txt`
3. Извлечение host:port из каждого конфига
4. Проверка: совпадает ли host/ip с whitelist?
5. Фильтрация несовпадающих конфигов

**Результат:**
- `bypass/` — SNI/CIDR + security-фильтрация
- `bypass-unsecure/` — только SNI/CIDR (без security)

## Верификация конфигов

### Двухуровневая архитектура

```
Raw-файлы → Верификация → Verified-файлы
  (raw/)                    (bypass-*.txt)
```

1. **Raw** — нетестированные конфиги, результат фильтрации без проверки
2. **Verified** — прошедшие верификацию, отсортированные по пингу

### Режимы верификации

| Режим | Механизм | Скорость | Точность |
|-------|----------|----------|----------|
| Xray-core | Запуск Xray → HTTP через SOCKS5 | 60-120 с | Высокая |
| TCP-ping | Asyncio TCP-соединение к host:port | ~5 с | Средняя |
| skip | Без проверки | 0 с | — |

### Пул портов

Для Xray-тестирования используется динамическое выделение портов в диапазонах:

| Диапазон | Назначение |
|----------|------------|
| 20000-21999 | Параллельное batch-тестирование |
| 22000-23999 | Прокси-цепочки (dialerProxy) |
| 24000-24999 | Персистентные прокси |

Каждый конфиг тестируется на уникальном порту, что исключает взаимное влияние.

## URL Health Report

Автоматический отчёт в конце каждого запуска:

- URL с 3+ последовательными ошибками загрузки удаляются из `URLS.txt`
- Конфиги из `servers.txt` с 3+ провалами верификации удаляются
- Топ источников по количеству рабочих конфигов
- Статистика сохраняется в `data/url_stats.json` (gitignored)

## Обработка ошибок

В парсерах Xray-конфигов каждое исключение логируется с контекстом. При ошибке парсинга конфиг не крашит тестирование — он просто отмечается как нерабочий и пропускается.

Потокобезопасность везде обеспечивается через `threading.Lock`.

## Защита от сбоев

Система использует circuit breaker для предотвращения каскадных отказов:

- `CIRCUIT_BREAKER_FAILURE_THRESHOLD = 5` — после 5 ошибок подряд breaker размыкается
- `CIRCUIT_BREAKER_RECOVERY_TIMEOUT = 60` — 60 с на восстановление
- Retry: 2 попытки на конфиг при Xray-тестировании, 5 попыток на GitHub upload
- **CIDR OOM guard:** `load_cidr_whitelist()` автоматически пропускает CIDR-диапазоны с >65536 адресов (предотвращает OOM от случайной /8 или /16 записи)

Временные файлы создаются с `TEMP_FILE_PERMISSIONS = 0o600` — только владелец имеет доступ.

## ManagedProcess — жизненный цикл подпроцессов

Управление Xray-процессами вынесено в `utils/managed_process.py`:

1. Процесс запускается с уникальным конфигом (`tempfile.mkstemp` + `chmod 0600`)
2. При завершении (успех, таймаут, ошибка) — гарантированная очистка временного файла
3. `ProcessRegistry` (`utils/process_registry.py`) отслеживает все запущенные процессы
4. При SIGINT/SIGTERM: `default_registry.cleanup(force=True)` завершает все процессы и восстанавливает proxy env vars

## Реестр процессов

Единый `ProcessRegistry` заменил три старых реестра:
- `_active_testers` (xray_tester.py)
- `_xray_process_registry` (ip_verifier.py)
- `_active_proxy_monitors` (proxy_monitor.py — пока остаётся отдельно, т.к. это thread-мониторы, а не процессы)

Сигнал-хендлер регистрируется только в `main.py` — xray_tester больше не регистрирует свой собственный SIGINT (предотвращает race condition).

## Валидация URL

Функция `is_valid_vpn_config_url()` проверяет, что строка является корректным VPN-URL:

```python
^(vmess|vless|trojan|ss|ssr|tuic|hysteria|hysteria2|hy2)://
```

Строки, не соответствующие ни одному протоколу, исключаются на этапе подготовки контента.

## Case-insensitive протоколы

Все парсеры и фильтры используют case-insensitive сопоставление протоколов. `VLess://`, `VMess://`, `TROJAN://` и т.д. корректно распознаются и обрабатываются. Ключи дедупликации нормализованы к нижнему регистру, поэтому `vless://host` и `VLess://host` дают одинаковый dedup-ключ.

## VPNConfig dataclass

`utils/vpn_config.py` содержит типизированную иерархию датаклассов для всех поддерживаемых протоколов:

```python
class VPNConfig(Protocol):
    host: str
    port: int
    def to_xray_outbound(self, tag: str) -> Dict: ...
    def to_url(self) -> str: ...
    def dedup_key(self) -> Tuple: ...
```

Это позволяет избежать повторного парсинга URL (особенно для VMess с base64-кодированными JSON) и даёт единую точку расширения при добавлении нового протокола.
